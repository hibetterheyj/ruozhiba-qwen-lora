## 3. 显存压测 (Phase 2.4)

MVP 训练峰值仅 53.5 GB，存在大量显存余量。通过 `probe_batch_size.sh` 脚本进行小步快跑压测 (`max_steps: 15`)：

| Batch Size | 结果 | 说明 |
|-----------|------|------|
| 16 | ✅ Pass | — |
| 24 | ✅ Pass | — |
| 32 | ✅ Pass | 最大安全值 |
| 48 | ❌ OOM | 需分配 21.74 GiB，仅剩 18.03 GiB |

**结论**: BS=32 为 L20Z 80GB 单卡的安全极限 (Qwen3-4B + LoRA rank=8, cutoff_len=2048)。

### OOM 修复 (Phase 2.5)

正式训练时 BS=32 触发 OOM，原因是压测未包含 eval/wandb/checkpoint 的额外内存开销。

**修复方案**: 梯度累积保持有效 batch size 不变：

| 参数 | 修复前 | 修复后 | 有效 BS |
|------|--------|--------|---------|
| per_device_train_batch_size | 32 | 16 | — |
| gradient_accumulation_steps | 1 | 2 | 16×2=32 |

## 6. MVP vs 正式训练对比

| 指标 | MVP (R8, 3 epochs) | 正式 R8 (7 epochs) | 正式 R16 (7 epochs) |
|------|---------------------|--------------------|--------------------|
| Batch Size | 8 | 32 (16×2) | 32 (16×2) |
| 总步数 | 993 | 581 | 581 |
| All-time best eval_loss | 0.8820 | 0.8870 | **0.8859** |
| 最终 train_loss | 0.9594 | 0.7631 | 0.6776 |
| 训练时长 | 12m 51s | 28m 23s | 28m 21s |

**关于 MVP eval_loss 更低的讨论**:

MVP 的 eval_loss (0.8820) 低于正式训练的最优值 (0.8859)，看似矛盾但合理：

1. **不同的 eval split**: MVP 和正式训练使用相同 seed=42 但有效 batch size 不同 (8 vs 32)，LLaMA-Factory 的随机 split 可能产生不同的 eval 子集
2. **不同的 eval 间隔**: MVP 每 100 步 eval (每 0.3 epoch)，正式训练也是每 100 步但 step 对应更多样本。更频繁的 eval 可能捕获到更低的瞬时 eval loss
3. **更多步数**: MVP 993 步 vs 正式 581 步，虽然 epoch 少但步数多（因 BS 更小），模型有更多更新机会
4. **eval_loss 差异仅 0.0039**，在统计噪声范围内，不构成实质性差异

### 9.3 综合分析

1. **数据量减少 63% → eval_loss 升高 8.6~11%**：R16 受影响较小 (+8.6%) 而 R8 受影响更大 (+11.0%)，说明更高 rank 在数据稀缺时有更好的特征提取能力
2. **训练时长线性缩放**：数据量减少 63%，训练时长也减少约 63% (28m → 10.5m)，吞吐量几乎不变
3. **R16 优势在小数据集上更明显**：全量时 R16 仅比 R8 好 0.12%，last3 时 R16 比 R8 好 2.3%——数据量越少，模型容量的优势越大
4. **eval_loss 不可直接跨数据集对比**：两组实验使用不同的 eval 子集（不同样本数量和分布），eval_loss 差异既反映模型能力差异，也包含数据分布差异。需通过统一测试集 (如 CQIA) 进行公平对比

## 10. 关键结论与后续建议

### 关键结论

1. **数据质量验证**: 2,785 条 (全量) 和 1,025 条 (近三年) ShareGPT 格式训练数据均能有效训练 Qwen3-4B 学习幽默分类任务，loss 正常收敛
2. **全量数据最优配置**: LoRA rank=16, alpha=32, epoch ~5，在拟合能力和泛化之间取得最佳平衡 (eval_loss = 0.8859)
3. **近三年数据实验**: R16_last3 在 7 epochs 内 eval_loss 持续下降至 0.9623，未出现过拟合，但绝对 eval_loss 高于全量数据，反映训练样本不足
4. **数据量影响**: 全量 (2,645 train) vs 近三年 (973 train)，训练数据减少 63% 导致 eval_loss 升高约 8.6% (0.8859 → 0.9623)，数据量对模型泛化能力影响显著
5. **过拟合边界**: 全量 rank=16 在 epoch 5 后过拟合；近三年数据量更小，但 7 epochs 内两种 rank 均未过拟合（eval 采样点较少，可能未捕获到拐点）
6. **显存利用**: 通过梯度累积 (BS=16×2=32)，在 L20Z 80GB 单卡上稳定训练

### 后续建议 (Phase 3)

1. **定量评估**: 使用 CQIA 测试集 (240 条) 计算 BLEU/ROUGE 等指标
2. **Before/After 对比**: 同一段子输入基座模型和微调模型，对比输出质量
3. **LLM-as-Judge**: 使用 Claude/GPT4 对比评判两个模型的分类准确性
4. **人工评估**: 随机抽样 50 条，人工判断分类准确性和思考过程质量
5. **全量 vs 近三年对比**: 对同一批测试样本，比较全量模型和近三年模型的输出质量，评估数据时效性 vs 数据量的权衡
